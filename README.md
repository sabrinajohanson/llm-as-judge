# llm-as-judge

Uses an LLM to evaluate the quality of test cases generated by another LLM, scoring each one across 5 objective criteria and producing a pass/fail verdict with justification.

[![CI (mock)](https://github.com/sabrinajohanson/llm-as-judge/actions/workflows/ci-mock.yml/badge.svg)](https://github.com/sabrinajohanson/llm-as-judge/actions/workflows/ci-mock.yml)

[![CI (live)](https://github.com/sabrinajohanson/llm-as-judge/actions/workflows/ci-live.yml/badge.svg)](https://github.com/sabrinajohanson/llm-as-judge/actions/workflows/ci-live.yml)

This project consumes the output of [`llm-test-case-generator`](https://github.com/sabrinajohanson/llm-test-case-generator) as its input, closing the loop between generating and evaluating AI-produced QA artifacts.

## Tech Stack

- Python
- OpenAI API (`gpt-4o`)
- python-dotenv

## How It Works

1. A list of test cases is read from `examples/input_test_cases.json` (a snapshot of output from `llm-test-case-generator`, including 3 intentionally flawed test cases used as control cases - see "Findings" below).
2. Each test case is sent to the OpenAI API individually, one API call per test case, with a prompt instructing the model to act as an impartial judge.
3. The judge scores the test case from 1 (very poor) to 5 (excellent) across 5 criteria:
   - **clarity** - is the test case unambiguous and executable without guessing?
   - **step_validity** - do the steps actually test what the title/description promise?
   - **verifiable_expected_result** - is the expected result objectively checkable, or vague?
   - **layer_correctness** - is the unit/integration/e2e classification consistent with what the test actually does?
   - **tbd_appropriateness** - is the `"TBD"` placeholder used only where genuinely justified?
4. An `overall_verdict` of `"pass"` or `"fail"` is derived: if any single criterion scores 1 or 2, the verdict is automatically `"fail"`, regardless of the other scores.
5. Results are saved to `output/evaluation_results.json`.

## Setup

1. Clone this repository.
2. Create a virtual environment and activate it.
3. Install dependencies: pip install -r requirements.txt
4. Create a `.env` file in the project root with your OpenAI API key: OPENAI_API_KEY=your_key_here
5. Run the judge: python main.py

## Findings

**1. Control cases: testing whether the judge actually judges**

A model that always says "pass" isn't a judge, it's a rubber stamp. To validate the judge was applying real criteria (not just approving everything), 3 intentionally flawed test cases were added to the input as controls:

| ID | Intentional flaw | Detected? |
|---|---|---|
| `TC-BAD-001` | Vague, unverifiable expected result + steps that don't complete what the title promises | ✅ Correctly failed (`step_validity`: 1, `verifiable_expected_result`: 1) |
| `TC-BAD-002` | Classified as `unit` despite involving cart + checkout + external service | ✅ Correctly failed (`layer_correctness`: 1) |
| `TC-BAD-003` | Classified as `unit` despite being a full user journey (login → checkout → confirmation) | ✅ Correctly failed (`layer_correctness`: 1) |

All 3 control cases were correctly identified as failing, confirming the judge applies the scoring criteria meaningfully rather than defaulting to positive verdicts.

**2. Unexpected discovery: the judge found real flaws in the original generator's output**

Beyond the intentional control cases, the judge also flagged two *original*, unmodified test cases from `llm-test-case-generator` (`TC-002` and `TC-008`, both login-related tests) as `fail`, citing incorrect `layer_correctness`: both were classified as `unit`, but the judge argued that testing login involves interaction with an authentication system and should be `integration` or `e2e` instead.

This was not planned, it surfaced during normal evaluation of the "good" test cases. It validates a limitation that was already noted (but not pinpointed) in `llm-test-case-generator`'s README: layer classification is not fully reliable, especially for cases involving user authentication. This project provided concrete, automated evidence of exactly which cases were affected.

**3. Why one API call per test case, instead of one call for the whole batch**

Evaluating test cases individually (rather than sending all of them in a single prompt) produces more focused, consistent judgments, at the cost of more API calls. For a small batch (11 test cases here), this tradeoff is worth it, for much larger batches, batching strategies would need to be reconsidered.

**4. Mutation testing reveals shallow assertion depth despite 100% test pass rate**

Running mutation testing (`mutmut`) against `judge.py` initially generated 76 mutants, of which only 16 were killed by the existing test suite, a mutation score of ~21%. This is a meaningfully different signal than the 100% pass rate of the 5 existing unit tests: passing tests confirm the code behaves correctly on the paths it exercises, while mutation testing reveals how thoroughly those paths are actually checked.

Reviewing a representative sample of survivors showed two categories:

- **Genuine gaps**: mutants like `open(path, "r", encoding=None)` (instead of `"utf-8"`) and `model=None` (instead of `"gpt-4o"`) survived because the tests never asserted on *how* dependencies were called, only on the final result. A silent regression removing the encoding or swapping the model would have gone undetected.
- **Cosmetic/low-value mutants**: mutants like `json.dumps(test_case, indent=None)` (instead of `indent=2`) or `print(None)` (instead of a formatted log line) survive because they don't affect actual behavior. Chasing these would test implementation details, not correctness.

Two tests were strengthened to close the genuine gaps: one asserting the exact arguments passed to `open()`, and one asserting the `model` parameter sent to the OpenAI API call. This raised the mutation score to ~32% (24/76 killed), closing every survivor related to file encoding and the model parameter, without chasing the cosmetic mutants identified above.

**Takeaway:** a 100% pass rate says nothing about how deeply a test suite actually verifies behavior. Mutation testing is a much stronger signal, but it also requires judgment to separate meaningful survivors from noise; treating every surviving mutant as a bug to fix would waste effort on details that don't matter.

## Testing

This project uses a two-tier testing strategy to balance thorough validation with API cost control:

- **Mock tests** (`tests/`): Unit tests that mock the OpenAI API call, validating prompt building, JSON parsing (including the malformed-response error path), and the result-aggregation logic in `evaluate_all_test_cases`. These run automatically on every push via GitHub Actions, at no cost.
- **Live contract test** (`tests/live/`): A test that calls the real OpenAI API and validates the structural contract of the response (required score fields, valid score range, allowed `overall_verdict` values), plus the core business rule: any score of 1 or 2 must force a `"fail"` verdict. This runs only when manually triggered via GitHub Actions (`workflow_dispatch`), to control token spend.

Run mock tests locally:
```bash
pytest tests/ -v --ignore=tests/live
```

Run the live test locally (requires a valid `OPENAI_API_KEY` in `.env`):
```bash
pytest tests/live/ -v
```

**Mutation testing:** Run on-demand via GitHub Actions (`Mutation Testing` workflow, `workflow_dispatch`), using `mutmut` against `judge.py`. Generates an HTML report (total/killed/survived counts, grouped by function), uploaded as a downloadable artifact rather than published permanently, since it's run manually rather than on every push. See "Findings" above for the current mutation score and interpretation.

**Test reports:**
- The mock CI publishes a full [Allure Report](https://sabrinajohanson.github.io/llm-as-judge/) on every push, showing the detailed results of the mocked test suite.
- The live CI does not publish a public report, since it only runs on-demand. Its results (Allure report and a `results.json` summary) are available as downloadable artifacts on the corresponding [CI (live) run](https://github.com/sabrinajohanson/llm-as-judge/actions/workflows/ci-live.yml).

## Known Limitations

- The judge itself is an LLM, and its judgments, while criteria-based, are not infallible; they represent a structured opinion, not ground truth.
- Evaluating one test case per API call means evaluation time and cost scale linearly with the number of test cases.
- The judge and the generator use the same model (`gpt-4o`) family; using a different model as judge could reduce shared blind spots (a direction for future iteration).
